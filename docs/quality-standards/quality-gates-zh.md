# 质量门：发布前检查清单

在发布任何客户项目之前运行的快速检查清单。这些门确保专业质量而不减慢您的工作流程。

## 为什么质量门很重要

**对于您的业务：**
- 防止令人尴尬的错误
- 减少修订请求
- 维护专业声誉
- 证明您的[定价](../business-model/pricing-economics-zh.md)合理

**对于您的工作流程：**
- 交接前5-10分钟检查
- 捕获95%的常见问题
- 系统性方法防止遗忘项目
- 建立客户信任

**规则**：没有项目在不通过关键7的情况下发布。特定门取决于类型。

---

## 关键7（不可协商）

修复这些或不要发布。这些是交易破坏者。

### 1. 所有交互元素工作 ✅

**要测试什么：**
```
✓ 所有按钮触发操作
✓ 所有表单成功提交
✓ 所有链接转到正确的目的地
✓ 所有社交媒体图标正确链接（或被删除）
✓ 模态框打开和关闭
✓ 手风琴/下拉菜单展开和折叠
✓ 导航菜单在移动设备上工作
✓ 搜索工作（如果存在）
✓ 联系表单传递消息
```

**如何测试：**
- 单击每个按钮
- 填写并提交每个表单
- 打开每个模态框/下拉菜单
- 尝试每个链接

**常见失败：**
- 链接到"#"或example.com的社交图标
- 没有后端的联系表单
- 移动设备上损坏的导航
- 无法关闭的模态框

**另请参阅**：[设计一致性 - 功能元素](./design-consistency-zh.md#功能交互元素)

---

### 2. 移动响应（最小375px）✅

**要测试什么：**
```
✓ 无水平滚动
✓ 文本可读而无需缩放
✓ 按钮可点击（不太小）
✓ 图像正确缩放
✓ 导航工作（汉堡菜单功能）
✓ 表单在移动设备上可用
✓ 内容不溢出容器
```

**如何测试：**
```
1. Chrome DevTools（F12）
2. 切换设备工具栏（Ctrl+Shift+M）
3. 设置为375px宽度（iPhone SE）
4. 滚动整个页面
5. 测试所有交互
```

**快速移动检查：**
- 英雄部分适合而无需滚动
- 导航可访问
- CTA按钮可见和可点击
- 联系信息易于点击
- 表单具有适当的输入类型（tel、email）

**常见失败：**
- 固定宽度容器
- 文本太小
- 按钮太靠近
- 具有微小输入的表单
- 未优化的图像导致加载缓慢

**另请参阅**：[性能 - 移动优化](./performance-zh.md)

### 3. 存在Meta标签 ✅

**必需标签：**
```html
<!-- 页面标题（唯一，50-60个字符） -->
<title>业务名称 - 地点的服务</title>

<!-- 元描述（150-160个字符） -->
<meta name="description" content="具有价值主张的具体描述">

<!-- 视口（移动支持） -->
<meta name="viewport" content="width=device-width, initial-scale=1">

<!-- Open Graph（社交分享） -->
<meta property="og:title" content="业务名称 - 服务">
<meta property="og:description" content="简要描述">
<meta property="og:image" content="https://example.com/og-image.jpg">
<meta property="og:url" content="https://example.com">

<!-- 网站图标 -->
<link rel="icon" type="image/png" href="/favicon.png">
```

**如何验证：**
```
1. 查看页面源代码（Ctrl+U）
2. 检查<head>部分
3. 验证所有标签存在
4. 测试OG图像（在Facebook/LinkedIn预览中粘贴URL）
```

**快速验证：**
- 标题在浏览器标签页中显示（不是"Home"或空白）
- 网站图标出现（不是默认浏览器图标）
- 在社交媒体上分享链接 - 显示正确的图像/文本

**另请参阅**：[SEO指南 - Meta标签](./seo-zh.md)

### 4. 无占位符内容 ✅

**要搜索什么：**
```
搜索整个代码库：
✓ "lorem ipsum"
✓ "test"
✓ "sample"
✓ "TODO"
✓ "placeholder"
✓ "["（括号表示占位符）
✓ "example.com"
✓ "your-business-name"
```

**如何搜索：**
```bash
# 搜索HTML文件
grep -r "lorem\|TODO\|test\|sample" .

# 或使用IDE的搜索（Ctrl+Shift+F）
```

**要删除的常见占位符：**
- Lorem ipsum文本
- "测试标题"或"示例部分"
- example@example.com
- (555) 555-5555
- "在此处输入您的业务名称"
- 生产代码中的TODO注释
- 测试图像（image-placeholder.jpg）

**检查清单：**
- [ ] 所有文本都是真实内容
- [ ] 所有联系信息准确
- [ ] 所有图像都是真实的（无占位符）
- [ ] 所有业务名称/地址正确
- [ ] 没有开发者注释可见

### 5. 异步操作上的加载状态 ✅

**什么需要加载状态：**
```
✓ 表单提交
✓ "联系我们"按钮
✓ 新闻通讯注册
✓ 登录/身份验证
✓ 搜索功能
✓ "加载更多"按钮
✓ 任何API调用
```

**最小实现：**
```javascript
// 按钮变为禁用 + 显示"加载中..."
button.disabled = true;
button.textContent = '发送中...';

// 操作完成后
button.disabled = false;
button.textContent = '提交';
```

**如何测试：**
- 提交表单（观察按钮状态）
- 触发任何异步操作
- 验证视觉反馈立即出现

**常见失败：**
- 没有反馈的表单提交
- 提交期间按钮保持可点击
- 没有进度指示
- 用户多次点击（双重提交）

**另请参阅**：[设计一致性 - 加载状态](./design-consistency-zh.md#加载状态)

### 6. 一致的间距和对齐 ✅

**视觉扫描检查清单：**
```
✓ 部分顶部/底部有均匀内边距
✓ 网格中的卡片正确对齐
✓ 文本块一致对齐
✓ 按钮相同大小（除非有意）
✓ 图库中图像相同宽高比
✓ 没有视觉上"浮动"的元素
✓ 页面间页眉/页脚一致
```

**如何检查：**
```
1. 缩小到50%（Ctrl + -）
2. 扫描视觉节奏
3. 检查部分有呼吸空间
4. 验证网格对齐
5. 寻找间距不一致
```

**常见问题：**
- 随机内边距值
- 未对齐的网格项
- 部分挤在一起
- 不一致的外边距

**快速修复：**
使用系统中的设计令牌（[设计一致性 - 间距](./design-consistency-zh.md#间距和节奏系统)）

### 7. 真实或零推荐信 ✅

**选项：**

**选项A - 真实推荐信：**
```
✓ 真实人名（名和姓）
✓ 具体引用（不是通用的）
✓ 上下文（职位，地点或标识符）
✓ 真实照片（可选但首选）
```

**选项B - 无推荐信部分：**
```
✓ 完全删除部分
✓ 用其他信任信号替换：
  - 营业年限
  - 客户数量
  - 认证/许可证
  - 行业隶属关系
```

**如何验证：**
- 阅读每个推荐信 - 它是具体的吗？
- 检查名称 - 全名，不是"John D."
- 验证照片 - 真人还是库存？
- Google引用 - 它是否出现在其他网站上？

**危险信号（删除这些）：**
- 没有上下文的"优质服务！"
- AI生成的面孔
- 通用名称（"Sarah P."）
- 多个网站上的相同推荐信
- 库存照片推荐信

**规则**：一个真实推荐信 > 三个假推荐信 > 无推荐信

**另请参阅**：[设计一致性 - 假推荐信](./design-consistency-zh.md#假的或通用的推荐信)

## 性能门

发布前的快速性能检查。

### Lighthouse快速审计 ✅

**如何运行：**
```
1. Chrome DevTools（F12）
2. Lighthouse选项卡
3. 选择：性能，可访问性，SEO，最佳实践
4. 设备：移动
5. 点击"分析页面加载"
```

**最低分数：**
```
✓ 性能：85+
✓ 可访问性：90+
✓ 最佳实践：90+
✓ SEO：95+
```

**如果分数低于目标：**
- 检查[性能指南](./performance-zh.md)进行优化
- 常见快速修复：优化图像，删除未使用的CSS/JS

### 图像优化 ✅

**快速检查：**
```
✓ 所有图像在500KB以下（最好在200KB以下）
✓ 图像使用现代格式（WebP，AVIF）
✓ 图像有宽度/高度属性（防止布局偏移）
✓ 图像有alt文本
✓ 英雄图像首先加载（优先）
✅ 折叠以下图像延迟加载
```

**如何验证：**
```
1. DevTools中的网络选项卡
2. 重新加载页面
3. 检查图像大小
4. 注意加载顺序
```

**快速优化：**
```bash
# 转换为WebP（如果使用Cloudflare Pages，自动）
# 否则使用https://squoosh.app/
```

**另请参阅**：[性能 - 图像优化](./performance-zh.md)

### Core Web Vitals ✅

**检查这些指标：**
```
✓ LCP（最大内容绘制）：< 2.5s
✓ FID/INP（首次输入延迟）：< 100ms
✓ CLS（累积布局偏移）：< 0.1
```

**如何检查：**
- Lighthouse报告（显示所有三个）
- Chrome DevTools性能选项卡
- [PageSpeed Insights](https://pagespeed.web.dev/)

**常见CLS修复：**
- 为图像添加宽度/高度
- 为动态内容保留空间
- 不要在现有内容上方注入内容

**另请参阅**：[性能 - Core Web Vitals](./performance-zh.md)

## 常见门失败

### 门捕获的前10个问题

1. **链接到"#"的社交链接**（门1：交互元素）
2. **移动水平滚动**（门2：响应式）
3. **缺少元描述**（门3：元标签）
4. **页脚中的lorem ipsum**（门4：占位符文本）
5. **没有表单加载状态**（门5：加载状态）
6. **未对齐的网格项**（门6：间距）
7. **通用推荐信**（门7：推荐信）
8. **错误电话号码**（本地企业：联系信息）
9. **通用英雄副本**（副本门：英雄）
10. **巨大未优化图像**（性能：图像）

## 关键要点

### 门防止尴尬

**没有门：**
- 使用lorem ipsum发布
- 损坏的联系表单
- 错误的电话号码
- 非功能性按钮
- 不专业的外观

**有门：**
- 一致质量
- 专业交付
- 客户信心
- 更少修订

### 5分钟节省小时

**时间投资：**
- 5-10分钟：发布前运行门
- 2-4小时：修复客户交付后发现的问题

**数学**：门比交付后修复效率高20倍。

### 建立习惯，而不是检查清单

**5-10个项目后：**
- 门变得自动
- 您在构建时捕获问题
- 检查清单变得更快
- 质量成为默认

**目标**：内化关键7，这样您永远不会在没有它们的情况下发布。
