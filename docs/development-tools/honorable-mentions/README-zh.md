# 荣誉提及 🏆

本部分突出了为寻求免费或高性价比替代方案以实施vibecoding工作流程的开发人员提供卓越价值的工具。虽然这些工具可能没有我的主要工具栈那样的精致或功能完整性，但它们为在没有财务承诺的情况下开始提供了坚实的基础。

## IDE原生AI助手

- **[Kilo Code](kilocode-zh.md)** - 多功能多平台AI助手，具有智能模式（代码/架构/询问/调试/编排器）、灵活定价（免费/PAYG/BYOK）和双重支持（VSCode + CLI）。非常适合喜欢具有视觉反馈的以IDE为中心的工作流程的开发人员。

## 免费和高性价比工具

- **[Qwen Coder / Qwen CLI](qwen-coder-zh.md)** - 阿里巴巴的免费开源编码助手，支持MCP
- **[Gemini CLI](gemini-cli-zh.md)** - Google的CLI，具有慷慨的免费层级和强大的推理
- **[AmpCode](ampcode-zh.md)** - 免费层级编码辅助，支持MCP
- （已推广）**TRAE**已移至推荐工具：参见[TRAE](../recommended-tools/trae-zh.md)
- **[Octofriend](octofriend-zh.md)** - 开源CLI，具有无缝Synthetic.new集成、出色的MCP支持，没有速率限制头痛
- **[Ghostty](ghostty-zh.md)** - 快速、轻量级终端模拟器，作为系统默认和Warp的替代方案
- **[iTerm2](iterm2-zh.md)** - macOS的强大、可定制终端模拟器，具有AI集成。

## 原生集成工具

- **[GitHub Copilot](github-copilot-zh.md)** - $10获得300次提示 + 无限GPT-4o-mini的良好价值；对学生免费；VS Code原生，但对于重度vibecoding有限

## 我放弃的工具 — 诚实评估

这些是我在生产中个人使用并决定从我的工作流程中放弃的工具。我将它们包括在荣誉提及中，因为它们值得了解，我的经验可以帮助您做出明智的决定。

- **[OpenSpec CLI](openspec-cli-zh.md)** - 规范驱动开发框架（已替换：优秀的工具，但Clavix提供更快的基于CLEAR的PRD生成和更好的AI代理集成）
- **[Traycer.ai](traycer-zh.md)** - AI驱动的开发规划和验证（已放弃：太慢、昂贵、对幻觉感到沮丧）
- **[GitHub Speckit](github-speckit-zh.md)** - GitHub的原生规范工具（已放弃：对于快节奏的vibecoding来说太重，更喜欢Clavix进行PRD生成）
- **[Cline / Roo Code](cline-roo-zh.md)** - 用于基于GUI的vibecoding的VSCode插件，具有GLM/Synthetic.new集成（已放弃：不必要的复杂性、工具/MCP调用可靠性问题）

### 为什么分享放弃的工具？
这些工具不是失败——它们是有价值的产品，只是不适合我特定的vibecoding工作流程。通过分享基于真实生产经验的诚实评估，我可以帮助您：
- **避免昂贵的错误**：在投入金钱和时间之前从我的经验中学习
- **做出明智的选择**：了解不同方法之间的权衡
- **找到更好的替代方案**：发现更具成本效益或更快的解决方案
- **识别用例**：其中一些工具可能非常适合您的特定需求，即使它们不适合我

目标是透明度：不仅展示什么有效，还展示我尝试了什么以及为什么我继续前进。

## 实施建议

### 入门策略
1. **从Qwen CLI或Octofriend开始**：初学者最容易访问的选项
2. **添加Gemini CLI**：对于需要更强推理的项目
3. **探索AmpCode**：作为特定用例的补充工具
4. **毕业到主要工具栈**：当准备好进行专业开发时

### 工作流程集成
- 使用这些工具学习vibecoding原则
- 实验MCP服务器配置
- 开发个人编码模式和偏好
- 建立对AI辅助开发的信心

### 过渡规划
- 监控使用模式和限制
- 识别专业工作所需的功能
- 在有理由时规划迁移到高级工具的路径
- 维护免费工具用于学习和实验
