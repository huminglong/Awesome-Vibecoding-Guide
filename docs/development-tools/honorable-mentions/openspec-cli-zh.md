# OpenSpec CLI — 规范框架 📐

OpenSpec CLI是一个强大的规范驱动开发工具，将纯语言描述转化为全面的项目计划、任务和技术规范。它通过创建结构化、可编辑的文档来指导开发过程，同时保持对话式vibecoding方法，弥合了想法和实施之间的差距。

## 🎯 核心价值主张

### 规范驱动开发

OpenSpec的魔力在于它能够将用户描述转换为完整的开发框架：

1. **纯语言输入** → 用自然语言描述您想要构建的内容
2. **AI驱动的生成** → OpenSpec自动创建提案、任务和详细规范
3. **Markdown文档** → 所有内容都保存为结构化的.md文件，以便于编辑和审查
4. **自然语言交互** → 在整个过程中继续与您的代理进行vibecoding
5. **分阶段实施** → 大功能分解为可管理的阶段

**您的初始提示越具描述性，输出质量越好。**

### 通用集成

OpenSpec几乎可以与任何编码代理无缝协作：
- **Zed**（通过AGENTS.md文件）
- **Claude**（通过CLAUDE.md指令）
- **Droid CLI**（通过自定义指令文件）
- **任何可以遵循基于文件指令的AI编码代理**

## 🚀 主要用例

### 1. 从头开始功能开发

**工作流程：**
```
用户："我想构建一个全面的用户身份验证系统，具有社交登录、电子邮件验证、基于角色的访问控制和管理仪表板"

OpenSpec输出：
- 阶段1：数据库架构设计和核心身份验证
- 阶段2：社交登录集成（Google、GitHub等）
- 阶段3：电子邮件验证和密码重置
- 阶段4：基于角色的访问控制实施
- 阶段5：管理仪表板和用户管理
```

**结果**：5个阶段的15-20个详细任务，每个任务都有技术规范和实施指导。

### 2. 大规模项目规划

OpenSpec擅长将复杂项目分解为可消化的组件：

- **电子商务平台** → 支付处理、库存管理、用户账户、管理面板
- **SaaS应用程序** → 多租户、计费、分析、用户入职
- **移动应用程序后端** → API设计、数据库架构、身份验证、实时功能

### 3. 重构和系统升级

非常适合规划主要的架构更改：
```
"我需要从REST迁移到GraphQL，实现缓存层，并重新设计数据库架构"
```

### 4. 快速原型设计

快速生成MVP和概念验证的规范：
```
"我需要一个简单的任务管理应用程序，具有拖放、实时更新和团队协作功能"
```

## 💡 主要功能

### 1. 持久上下文管理

所有规范都保存为`.md`文件，提供：
- **版本控制** → 随时间跟踪规范的更改
- **上下文保留** → AI代理保持对项目范围的理解
- **易于审查** → 用于团队协作的人类可读文档
- **可编辑工作流程** → 根据需要修改任务和规范

### 2. 基于阶段的开发

OpenSpec自动将大功能分解为逻辑阶段：
- **基础阶段** → 核心架构和数据库设计
- **功能阶段** → 单个功能实施
- **集成阶段** → 连接组件和测试
- **完善阶段** → UI/UX改进和优化

### 3. 多LLM工作流程支持

OpenSpec最强大的功能之一是能够为不同的开发阶段启用不同的LLM：

**示例工作流程：**
```
阶段1：规划和规范
→ 使用GPT-5-high（通过Droid CLI）进行复杂的架构决策
→ 生成全面的计划和详细规范

阶段2：实施
→ 切换到GLM-4.6以进行高性价比开发
→ 按照批准的规范执行单个任务

结果：优质规划 + 预算友好的实施
```

**好处：**
- **成本优化** → 仅在高价值规划中使用昂贵的LLM
- **质量保证** → 为关键决策获得最佳思考
- **开发速度** → 使用经过验证的规范更快实施
- **可靠性** → 无论实施使用什么LLM，输出都一致

## 🔧 实际实施

### 逐步工作流程

1. **自然语言规范**
   - 只需用简单的英语向您的AI代理描述您的功能需求
   - 示例："我想构建一个具有社交登录和电子邮件验证的用户身份验证系统"
   - 您的描述越详细，生成的规范越好

2. **审查生成的规范**
   - 打开OpenSpec生成的`.md`文件
   - 审查提案、任务和技术规范
   - 根据需要编辑以符合您的要求

3. **基于阶段的开发**
   - 与您的AI代理一起完成每个阶段
   - 只需告诉您的代理您接下来想要处理哪个阶段
   - 不需要复杂的命令或语法

4. **迭代完善**
   - 通过自然对话在阶段之间修改规范
   - 根据开发见解调整任务
   - 在整个项目中保持上下文

### 最佳实践

1. **描述性输入很重要**
   - 包括业务需求、技术约束、用户场景
   - 提及性能要求、可扩展性考虑
   - 指定需要集成的现有系统

2. **实施前审查**
   - 在编码前始终审查生成的规范
   - 编辑任务以匹配您的编码风格和偏好
   - 根据您团队的工作流程调整阶段边界

3. **战略性地使用不同的LLM**
   - 高级LLM用于规划和复杂问题解决
   - 高性价比LLM用于常规实施
   - 需要时使用特定领域的专用模型

## 🎯 底线

OpenSpec CLI将模糊的想法转化为具体的、可操作的开发计划，同时保留自然、对话式的vibecoding体验。与需要记忆命令和语法的复杂CLI工具不同，OpenSpec让您继续用简单的英语与您的AI代理交谈，同时自动生成结构化规范。通过创建与任何编码代理一起工作的详细规范，它实现了可靠、可扩展的开发，同时保持为不同任务使用不同LLM的灵活性。结果是构建从简单功能到复杂企业应用程序的任何内容的系统方法，而不会干扰您的自然开发工作流程。
